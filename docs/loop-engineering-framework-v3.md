## 一、五阶段循环，和它现在挂在 template 上的具体机制

```
                    ┌─────────────────────────────────────────────┐
                    │                                               │
                    ▼                                               │
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ EXPLORE  │──▶│  SPEC    │──▶│  BUILD   │──▶│  SHIP    │──▶│ EVOLVE   │──┐
│ 多模型探讨│   │ /spec    │   │ Claude   │   │ /deploy  │   │ /status- │  │
│ →explor- │   │ →PRD/TDD │   │ Code 执行 │   │ →Cloud   │   │ report   │  │
│ ation.md │   │          │   │ (hooks   │   │ Run      │   │          │  │
│          │   │          │   │ 强制门禁) │   │          │   │          │  │
└──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
                                                                          │
                    ◀─────────────────────────────────────────────────┘
                         （EVOLVE 的产出回写 PROGRESS.md / DOMAIN.md，
                          作为下一轮 EXPLORE 的起点，不是从零开始）
```

每个阶段不再只是"你手动做的一件事"，而是 template 里有具体机制接管衔接点：

| 阶段 | 人做什么 | template 机制接管什么 | 这次实跑验证/修正了什么 |
|---|---|---|---|
| EXPLORE | 在 Cursor 里切换 Claude/GPT/DeepSeek/Gemini，讨论需求边界、技术路径、风险 | 无强制工具，但有**固定文件格式**：`docs/explorations/{date}-{slug}-exploration.md`，含核心问题/领域模型草案/技术可行性/管理层视角/模型意见汇总表/DECISION LOG | **验证**：多模型交叉验证确实抓出了单模型会漏的盲点（GPT 和 DeepSeek 独立提出同一个"生成前不能改"的产品缺口，互相印证比单一模型意见更可信）。**新增细节**：模型意见汇总表要按"问题"分行而不是按"模型"分行，否则交叉对比困难 |
| SPEC → BUILD 之间 | 确认 DECISION LOG 里的必选项 | `/init` 的 **Check 2** 会扫描 DECISION LOG，未全部 `[x]` 就拒绝继续，并给出具体缺口提示，不允许跳过 | **验证**：这道关卡这次真的拦下了一次"验收标准没定就想往下走"的情况（见下方"踩坑实录"）。机制按设计工作 |
| SPEC | 把探索结论收敛成 DOMAIN.md / CLAUDE.md PART 2 / smoke-test.sh | `/init` 四步生成流程（每步生成草稿 → 用户确认 → 才写入），完成后自动 commit；之后 `/spec` 另外生成 PRD/TDD 到 `docs/explorations/` | **验证**：`/init` 的 guardrail（"不能凭空填不可追溯的内容，要问用户"）这次真实触发——验收标准的四个量化指标没有依据，`/init` 没有自己编数字，而是停下来给出两个选项让用户选。这是这套机制里最关键的一条防线，运行如预期 |
| BUILD | 用 Claude Code 写代码 | `settings.json` 的 **PreToolUse hook**：CLAUDE.md 里如果还有 `/init 填充` 占位符，禁止写源代码文件；**PostToolUse hook**：每次写文件后自动跑 lint；**spec-reviewer subagent**：检查 PRD 和 DOMAIN.md 是否有可量化验收标准、ROI、术语一致性 | 这次还没跑到这一步，留待下次同步 |
| SHIP | 部署 | `/deploy [staging\|production]`：先跑 pre-deploy 检查，再 `deploy.sh`，再 `smoke-test.sh` 验证；不允许跳过 staging 直接上 production | 还没跑到 |
| EVOLVE | 多模型再评一遍 | `/status-report`：读 `PROGRESS.md` + `git log` + `git status`，生成结构化周报；**Stop hook** 在每次 session 结束时自动把 git diff 摘要追加进 `PROGRESS.md`，下次 session 不用全量扫 repo 找上下文 | 还没跑到 |

## 二、三层 Context 资产模型（这次实跑里验证的部分）

v2 设计稿提出的"逻辑底座 + 边界指令 + 安全绳"三层模型，这次实跑印证了它的分工边界确实清晰可执行：

```
┌─────────────────────────────────────────────────────┐
│  DOMAIN.md  —— 逻辑底座                                │
│  实体关系、业务规则、术语表、数据边界、状态机、外部系统    │
│  「业务规则段落由你拥有，agent 只能建议修改，不能自行修改」│
├─────────────────────────────────────────────────────┤
│  CLAUDE.md  —— 边界指令                                │
│  PART 1 全局规则（协作偏好/模型分配/代码规范/环境约束/    │
│  12条跨项目 learnings，所有项目继承，不在项目里改）       │
│  PART 2 项目专属（/init 填，不回流 template）            │
├─────────────────────────────────────────────────────┤
│  smoke-test.sh —— 安全绳                                │
│  从"验收标准"直接生成的 check() 断言，部署后立刻验证       │
└─────────────────────────────────────────────────────┘
```

这次实跑发现的关键依赖关系（v2 设计稿没明确写出来，是这次才看清的）：

**`smoke-test.sh` 的质量完全取决于"验收标准"这一项在 EXPLORE 阶段有没有量化。** 这是这次撞见的真实因果链：探索文件的 DECISION LOG 里"验收标准"留白 → `/init` Check 2 拦截 → 不拍板这四个数字，`/init` 没法往下走 → DOMAIN.md 业务规则段会是空的 → `smoke-test.sh` 的 `check()` 调用没有具体断言可写 → SHIP 阶段的部署验证形同虚设。也就是说，**这条链子的强度由最上游 EXPLORE 阶段那张"模型意见汇总表"愿不愿意逼出具体数字决定**，不是 BUILD/SHIP 阶段能补救的。

实操含义：EXPLORE 阶段问多模型意见时，**不要只问"对不对"，要追问"那具体数字应该是多少"**——这次 GPT/DeepSeek 都点出了"需要可读性验收标准"这个方向，但都没给具体数字（字数上限、超时秒数），最后还是要靠你自己拍板。多模型评审能找到**该问什么问题**，但**给最终数字**仍然是人的工作，不要指望模型替你拍板。

## 三、A/B/C 模型分配，结合这次 EXPLORE 实操微调

v2 原设计：
- **A（架构/探索）**：Claude/Gemini，在 Cursor 里用
- **B（核心构建）**：Claude Code（Sonnet），不可降级
- **C（样板/测试）**：DeepSeek，省 token

这次 EXPLORE 阶段实际用了三个模型（GPT、DeepSeek、Gemini）对同一组问题给意见，属于"A 模型"角色的并行化——不是只用一个 A 模型，而是**多个 A 模型交叉验证**。这次的经验显示这值得固化成模式：

**多模型 EXPLORE 的实操模式**：同一份探索文件草稿，分别粘给 2-3 个模型问"对既定路径的看法 + 未考虑到的风险"，汇总进同一张表（按问题分行，不按模型分行），自己对每行做判断后写入"你的判断"列。**不需要等所有模型都回复才能继续**——这次 Gemini 的回复一直没传上来，但探索文件先用 GPT+DeepSeek 的意见定稿推进，Gemini 的意见随时可以再追加，不构成阻塞。多模型评审是增量过程,不是同步关卡。

## 四、这次新增的一条 Loop 原则：分支/仓库状态要先验证，不要假设

这是这次踩的弯路，值得写进框架，下次直接跳过：

**问题**：抓取 GitHub 仓库时，假设 README 里出现的分支名（或者之前对话提到的分支）就是当前的默认分支状态，结果抓到的是一个早期、空的非默认分支，导致基于错误前提重写了一遍其实已经完整存在的 `/init` skill 和 CLAUDE.md 内容。

**修正后的原则**：核对仓库现状时，**优先要求拿到 zip 包或者实际文件内容**，而不是反复用网页抓取工具去猜分支/查看 GitHub 页面——网页抓取容易被导航噪音吃掉 token、容易抓到非默认分支、还可能被 robots.txt 挡掉。一次"麻烦你导出 zip 发我"比连续五次网页抓取更省时间也更准确。

**这条原则在框架里的位置**：属于 EXPLORE 阶段开始前的"环境核实"步骤，应该在种子问题模板里加一条隐含检查——**任何涉及"现有代码/仓库状态"的探索，先拿到真实文件内容，再开始设计**，不要在未经验证的假设上往前推。
